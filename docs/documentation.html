Documentation

Everything you need to run, configure, and extend Engram.

Installation

Engram is a Swift package. Clone and build:

git clone https://github.com/anthropics/engram.git
cd engram
swift build -c release
cp .build/release/engram /usr/local/bin/

Requirements: macOS 14 (Sonoma) or later. Swift 5.9 or later. Xcode command-line tools.

The only external dependency is swift-argument-parser for the CLI. Everything else uses Foundation and native macOS frameworks.

Quick Start

# Option 1: API key
export ANTHROPIC_API_KEY="sk-ant-..."
engram chat

# Option 2: Interactive setup
engram setup

# Option 3: OAuth (use your Claude Pro subscription)
engram login

That's it. The agent will create ~/.engram/ on first run with identity files, memory directory, and default configuration.

Configuration

All state lives in ~/.engram/:

~/.engram/
  config.json       ← model, provider, endpoints, MCP servers
  .env              ← API keys (file permissions 0600)
  SOUL.md           ← agent personality and name
  USER.md           ← what the agent knows about you
  BOOTSTRAP.md      ← first-run behavior (auto-deleted)
  memory/           ← nugget JSON files
  sessions/         ← conversation JSONL files
  skills/           ← installed skills
  cron/              ← scheduled job definitions

config.json

{
  "model": "claude-opus-4-6",
  "summaryModel": "claude-haiku-4-5-20251001",
  "provider": "anthropic",
  "maxTokens": 8192,
  "maxIterations": 50,
  "contextWindow": 200000,
  "memoryDir": "~/.engram/memory",
  "sessionDir": "~/.engram/sessions",
  "mcpServers": {},
  "gateway": {}
}
FieldDescription
modelPrimary model ID. Any model supported by the configured provider.
summaryModelCheap model used for context compaction. Falls back to primary model if null.
provideranthropic, openai, or codex. Determines API format.
baseURLCustom API endpoint. Use this for OpenRouter, Ollama, or self-hosted inference.

.env

API keys are stored in ~/.engram/.env with restrictive file permissions (0600). The format is standard dotenv:

ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
OPENROUTER_API_KEY=sk-or-...

Environment variables set in your shell take precedence over the .env file.

First Conversation

On the first run, Engram reads BOOTSTRAP.md and behaves accordingly: it asks your name naturally, learns how you communicate, and begins building context. It will update USER.md with what it learns and delete BOOTSTRAP.md once it knows who you are.

You don't need to configure the agent's personality. But if you want to, edit SOUL.md directly or tell the agent to change its name or behavior.

Memory

Engram uses Holographic Reduced Representations (HRR), a model from cognitive science. Instead of storing facts in rows or embedding them in a high-dimensional space, HRR encodes key-value pairs as bound complex vectors and superposes them into a single holographic trace.

How it works

  1. Vector generation. Each unique string maps to a deterministic 512-dimensional complex vector via a seeded PRNG (Mulberry32 with FNV-1a hashing). The same string always produces the same vector.
  2. Binding. A key-value pair is fused by element-wise complex multiplication (circular convolution in the frequency domain). This produces a single vector that encodes the association.
  3. Superposition. Multiple bindings are added together into one memory vector. Information from different facts occupies the same space, like holograms layered on film.
  4. Recall. To retrieve a value, multiply the memory vector by the conjugate of the key vector. The result is a noisy approximation of the value vector. Compare it against known values by cosine similarity to identify the match.
// Store
let keyVec = ComplexVector.random(for: "key:favorite_color", dimension: 512)
let valVec = ComplexVector.random(for: "val:black", dimension: 512)
memory = memory.add(keyVec.bind(with: valVec))

// Recall
let decoded = memory.unbind(with: keyVec)
let similarity = decoded.cosineSimilarity(with: valVec) // → ~0.97

Recall is algebraic — a fixed number of multiplications and additions. No search algorithm, no nearest-neighbor index, no API call. In-process, sub-millisecond.

Nuggets

A nugget is a topic-scoped holographic memory. The agent organizes facts into nuggets like preferences, project, people. Each nugget is an independent complex vector with its own fact list.

The Shelf manages all nuggets. When you query without specifying a nugget, the shelf searches all of them and returns the highest-confidence match.

Each fact tracks a hit count — the number of times it has been recalled across distinct sessions. This drives promotion.

Fact Promotion

A fact recalled three or more times across separate sessions promotes into the system prompt as permanent context. Promoted facts are injected at the start of every conversation, so the agent doesn't need to recall them — it already knows.

This creates a natural learning loop: the agent uses what's useful, and what's useful becomes foundational knowledge. Facts that were stored but never recalled stay in the hologram, addressable but inert.

Identity Files

FilePurposeLifecycle
SOUL.mdAgent name, personality, behavioral guidelinesPersistent. Updated by agent or user.
USER.mdUser profile — name, timezone, preferences, projectsPersistent. Built incrementally by the agent.
BOOTSTRAP.mdFirst-run instructionsTemporary. Auto-deleted after first conversation.

These are plain markdown with no special schema. The agent reads them at the start of every session and updates them silently as it learns. You can edit them directly at any time.

Memory is invisible to the user. The agent never announces that it's remembering, recalling, or updating identity files. It just knows things. This is enforced in the system prompt.

Agent Loop

The core execution model is a standard receive → think → act → observe loop:

  1. User input is appended to message history.
  2. Context is compacted if needed (see below).
  3. History + system prompt + tool definitions are sent to the LLM.
  4. If the response contains tool calls, execute them and append results.
  5. Repeat until the LLM stops calling tools or the iteration limit is reached.

The iteration limit (default: 50) prevents runaway tool loops. Each tool result is appended to the session file for persistence.

Context Management

The ContextManager tracks token usage and triggers compaction when the context window reaches 50% utilization.

Compaction works by:

  1. Keeping the first N messages (identity context) and last M messages (recent work).
  2. Summarizing the middle section using a cheap model (configured via summaryModel).
  3. Replacing the middle with a single summary message.

This allows arbitrarily long conversations without hitting context limits. The agent transparently manages its own window.

Sessions

Each session is a JSONL file in ~/.engram/sessions/. Every message is a JSON object on one line, forming a tree via parent IDs.

session_20260320_141523_a7b2c9d1.jsonl

Sessions support:

  • Resume: The --resume flag continues the most recent session.
  • Compaction: Old sessions can be compacted with summaries.
  • Full-text search: FTS5 indexing across all session files (see Session Search).

One-Shot Mode

Pass a prompt as an argument to get a single response:

engram chat "What meetings do I have today?"
engram chat "Find all TODO comments in this project"
engram chat "Summarize the last 3 git commits"

The agent has full tool access in one-shot mode. It can read files, run commands, check your calendar, and return a synthesized answer.

Tools Overview

Tools are registered at startup via ToolRegistry. Each tool implements the Tool protocol: a name, description, JSON schema for inputs, and an async execute method. The agent selects and invokes tools autonomously based on the user's request.

Tool definitions are sent to the LLM in the provider's native format (Anthropic tool_use or OpenAI function_calling). The agent can chain multiple tool calls in a single turn.

Skills

Skills are markdown files with YAML frontmatter that provide instructions the agent can load on demand. They're the primary way to extend Engram's behavior without writing Swift code.

Skills are discovered from:

  1. ~/.engram/skills/ — user-level skills
  2. .engram/skills/ — project-level skills (walking up from cwd)

Skill Format

A skill can be a single markdown file or a directory:

~/.engram/skills/my-skill/
  SKILL.md          ← instructions (required)
  references/       ← supporting documents
  templates/        ← output templates

SKILL.md uses YAML frontmatter:

---
name: my-skill
description: What this skill does in one line
version: 1.0.0
auto_load: false
tags: [coding, review]
---

Instructions for the agent when this skill is active.

You can reference files in the references/ and templates/
directories — the agent can read them via skill_view.
FieldDescription
nameUnique skill identifier. Lowercase, hyphens allowed.
descriptionOne-line description shown in skill_list output.
versionSemantic version. Informational.
auto_loadIf true, skill instructions are injected into every system prompt.
tagsComma-separated or bracketed list for categorization.

Installing Skills from GitHub

# Full URL
engram skills install https://github.com/user/repo

# Shorthand
engram skills install user/repo

# Subdirectory of a monorepo
engram skills install user/repo#my-skill

Skills are cloned into ~/.engram/skills/ and immediately available. The .git directory is removed to save space.

Creating Skills at Runtime

The agent can create skills through the skill_create tool — it literally extends its own capabilities during a conversation. Created skills persist across sessions.

Example: after helping you with a complex code review, the agent might create a "code-review" skill encoding the patterns and standards it used, so it can apply them consistently in future reviews.

Injection protection. Skills are scanned for prompt injection patterns before loading. Content containing phrases like "ignore previous instructions" or attempts to exfiltrate API keys is rejected.

Gateway

The gateway system lets you talk to Engram from messaging platforms. Each platform implements the Platform protocol: start, stop, poll for messages, send text, send files. The daemon polls all enabled platforms on a 5-second interval.

# Configure platforms
engram gateway

# Start the daemon to begin listening
engram daemon install
engram daemon start

Messages from any platform go through the same agent loop with the same memory, tools, and skills. The agent knows which platform a message came from and can send responses back through it.

iMessage

Sends messages via AppleScript. Reads incoming messages by polling ~/Library/Messages/chat.db (SQLite).

Requirements:

  • Full Disk Access for the engram process (System Settings → Privacy → Full Disk Access).
  • An active Messages session.

Configuration:

{
  "gateway": {
    "imessage": {
      "enabled": true,
      "allowedSenders": ["+1234567890"]
    }
  }
}

Telegram

Uses the Bot API with long-polling (getUpdates). Supports file uploads with automatic content-type detection (audio vs document).

Setup:

  1. Create a bot via @BotFather.
  2. Get the bot token.
  3. Configure:
{
  "gateway": {
    "telegram": {
      "enabled": true,
      "token": "123456:ABC-DEF..."
    }
  }
}

Messages longer than 4,000 characters are automatically split.

Discord

Connects via WebSocket gateway with heartbeat management. Listens for MESSAGE_CREATE events.

Setup:

  1. Create an application at the Discord Developer Portal.
  2. Create a bot, copy the token.
  3. Enable Message Content Intent.
  4. Invite the bot to your server.
{
  "gateway": {
    "discord": {
      "enabled": true,
      "token": "...",
      "channelId": "123456789"
    }
  }
}

Slack

Uses Socket Mode (WebSocket) with the Web API for sending messages. Requires both an app-level token and a bot token.

Setup:

  1. Create an app at api.slack.com.
  2. Enable Socket Mode. Generate an app-level token (xapp-).
  3. Add bot scopes: chat:write, channels:history, channels:read.
  4. Install to workspace. Copy bot token (xoxb-).
{
  "gateway": {
    "slack": {
      "enabled": true,
      "appToken": "xapp-...",
      "botToken": "xoxb-...",
      "allowedChannelIds": ["C01234567"]
    }
  }
}

Slack messages are filtered by channel allowlist. Edits and join events are ignored.

Email

Polls IMAP for unseen messages. Sends replies via SMTP. Uses curl — no external dependencies.

{
  "gateway": {
    "email": {
      "enabled": true,
      "imapHost": "imap.gmail.com",
      "imapPort": 993,
      "smtpHost": "smtp.gmail.com",
      "smtpPort": 587,
      "username": "you@gmail.com",
      "password": "app-specific-password"
    }
  }
}

For Gmail, use an App Password (not your account password). Polls up to 5 unseen messages per tick.

Home Assistant

Connects to Home Assistant via REST API. Sends persistent notifications and can call services for automation.

Daemon Mode

The daemon runs Engram as a macOS LaunchAgent — a background process that survives reboots and auto-restarts on crash.

engram daemon install    # Create LaunchAgent plist
engram daemon start      # Start the daemon
engram daemon status     # Check if running (PID, uptime)
engram daemon stop       # Stop the daemon
engram daemon logs       # Tail stdout and stderr

The daemon loop runs on multiple intervals:

IntervalAction
5 secondsPoll all enabled gateway platforms for incoming messages.
60 secondsTick the cron scheduler. Save memory. Evict idle sessions.
5 minutesReconnect any disconnected gateway platforms.
6 hoursMemory consolidation pass.

Sessions idle for more than 1 hour are automatically closed. The LaunchAgent plist is installed to ~/Library/LaunchAgents/ with KeepAlive and a 5-second throttle interval.

Scheduling (Cron)

Schedule prompts to run on a cron expression. The cron scheduler ticks every 60 seconds in the daemon loop.

# The agent can create cron jobs via tools:
cron_create: {
  "name": "morning-briefing",
  "schedule": "0 9 * * 1-5",
  "prompt": "What's on my calendar today? Any emails I should know about?"
}

Standard 5-field cron syntax: minute, hour, day-of-month, month, day-of-week. Supports wildcards (*), steps (*/15), ranges (1-5), and lists (1,3,5).

Jobs are persisted to ~/.engram/cron/jobs.json. Each job has an ID, enable/disable toggle, and last-run timestamp for deduplication.

MCP (Model Context Protocol)

Engram supports MCP for connecting external tool servers. MCP servers are subprocess-based — Engram launches them, communicates via JSON-RPC 2.0 over stdin/stdout, and wraps discovered tools into the agent's tool registry.

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/you/projects"]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://localhost/mydb"
      }
    }
  }
}

MCP tools appear alongside built-in tools with no special handling. The agent doesn't know the difference.

OAuth

Engram supports two OAuth flows:

Anthropic (Claude Code)

PKCE-based browser flow. If you have Claude Code installed, Engram can read existing credentials from the macOS Keychain. Otherwise it launches the full OAuth flow and stores tokens at ~/.engram/oauth_credentials.json with 0600 permissions.

OpenAI (Device Code)

Displays a user code and URL. You authenticate in a browser, and Engram polls for the token. Works with ChatGPT Plus and Pro subscriptions via the Codex Responses API.

engram login    # Interactive — pick a provider and authenticate

Tokens are refreshed automatically before they expire. The LLMClient checks expiration before every request.

Sub-Agents

The delegate tool spawns an isolated sub-agent with its own ToolRegistry, SessionManager, and conversation history. This is useful for:

  • Parallel research tasks
  • Sandboxed operations that shouldn't pollute the main context
  • Long-running work that would consume too much of the context window

Sub-agents share the same Shelf (memory) and LLMClient, but have independent tool registries and sessions. Results are truncated to 4KB before returning to the parent agent.

Session Search

Engram indexes all past conversations into a SQLite FTS5 virtual table, enabling full-text search across your entire conversation history.

The session_search tool takes a query string and returns matching messages with timestamps, session file references, and highlighted match excerpts. The index is rebuilt automatically when new session files are detected.

// The agent can search its own history:
session_search: { "query": "deployment strategy for the API" }

This is distinct from holographic memory — session search finds exact text in past conversations, while memory recall finds semantic associations between facts.
